Diseño Del Experimento Sprint 04

En esta sección deseamos determinar si las decisiones de arquitecturas tomadas hasta el momento van por el camino que deseamos, para esto presentaremos el diseño del experimento del sprint 04 que está enfocado en el atributo de calidad de seguridad.

Diseño Del Experimento

Pruebas De Estilos Y Tácticas De Disponiblidad Proyecto NIDOO
Propósito Este experimento pretende satisfacer las necesidades de los usuarios definidas en las ASRs 08, 13, 23 y 24 que a través de las tácticas definidas para el diseño de la arquitectura favorezca la seguridad de los componentes del sistema en cuanto a integridad, disponibilidad, confidencialidad y confidencialidad.
Resultados esperados Se espera que al finalizar el experimento se evalué el funcionalmente las decisiones de diseño para satisfacer la seguridad de los componentes de Nidoo minimizando el tradeoff entre los atributos de calidad implementados en los sprint anteriores.

También se espera que el sistema sea capaz de detectar, resistir, reaccionar a ataques e iniciar un procedimiento de recuperación. Para esto se probarán distintos ataques en el experimento para evaluar detalladamente el comportamiento de los componentes y si es necesario afinar la arquitectura hasta tener la mejor versión posible.

Analizar el comportamiento de los microservicios al generar un ataque que comprometa la integridad en cuanto la modificación de tarifas de parqueadero. Para esto, de manera intencional y simulando un atacante, cambiaremos una tarifa de parqueadero sin ser autorizados y verificar si el sistema es capaz de revocar el ataque y notificar los stakeholders delegados y si es necesario.   • Api gateway
  • Authorization server
  • Heroku firewall
  • Auditoria (MongoDB)
Elementos de arquitectura involucrados Los nuevos elementos de arquitectura involucrados para aplicar tácticas de seguridad y así mismos la integridad, confidencialidad y disponibilidad se decidió utilizar un servidor de autorización, el cual estará encargado de la gestión de tokens para la autorización de usuarios. En Heroku firewall se definirán reglas específicas de acceso a través de una sola IP y puerto para garantizar una sola vía de acceso por microservicio. Un api gateway (Kong), el cual estará encargado de recibir solicitudes con protocolos seguros (TLS, SSL) y direccionarlos a los respectivos microservicios. Se definirán disparadores (Triggers) en MongoDB, los cuales se ejecutarán cuando se generen eventos de tipo on-Update, on-Delete y se almacenarán en una tabla para mantener una bitácora de los eventos del microservicio.
